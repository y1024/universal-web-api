🔧 维护须知 & 使用预期

1️⃣ 站点突然不能用了？优先检查选择器

网站改版后，最常失效的通常是选择器，也就是告诉脚本“输入框、发送按钮、结果容器分别在哪”的那组 CSS 规则。大多数站点突然不能用时，第一反应应该先检查选择器，而不是先怀疑解析器。

选择器更新的常见做法：

  1. 在受控浏览器里对你想点击或想读取的元素右键，选择“检查”。
  2. 把对应的元素树截图或复制出来。
  3. 把这段元素树发给 AI，让 AI 帮你写出新的选择器。
  4. 回到控制台，把新的选择器粘贴替换进去后再测试。

解析器问题会麻烦一些：如果不是选择器的问题，才更可能是提取器 / 解析器失效。这通常需要拦截网络请求、查看响应结构，再继续分析，难度会比改选择器高得多。

2️⃣ 各站点的维护覆盖范围

Gemini、deepseek 是作者日常使用最多、维护也最及时的站点。其它站点并不保证第一时间跟进；如果遇到问题，更推荐加 QQ 群反馈，通常会比单纯提 Issue 更快拿到回应。

3️⃣ Function Calling 目前还不稳定

Function Calling 这块目前较为依赖 AI 本身的理解能力。可能仍然存在一定问题。

4️⃣ 遇到问题，优先让 AI 帮你分析

选择器失效、配置写错、工作流顺序不对，这类问题很多时候不需要等作者处理。把现象、日志截图、元素树交给 AI，通常会比单纯等人工回复更快。

5️⃣ 反馈 Bug 的正确姿势

反馈时尽量描述清楚你做了什么、期望看到什么、实际看到了什么，并附上 DEBUG 级别日志。如果你觉得自己表述不够清晰，可以先让 AI 帮你整理一版再发。

6️⃣ 如果你自己修好了，也欢迎把配置分享回来

如果你已经把某个站点修好了，不管是更新了选择器，还是补好了配置文件，都非常欢迎把改动分享给作者或群友。这样能让整个社区都少走一遍弯路。

7️⃣ 项目的核心场景

本项目最初主要是围绕酒馆 / RP（角色扮演）场景设计和打磨的，作者的精力也主要投在这个方向。其它场景不是不能用，但维护上更偏向“能用就用，不保证持续深耕”。

📌 先搞清一件事：脚本启动后通常会出现两个浏览器窗口。受控浏览器由脚本自动启动，用来登录 AI 网站并真正收发消息；你自己的普通浏览器或教程页窗口只负责看文档、日常上网。脚本运行时请不要手动操作受控浏览器，窗口可以最小化，但不要缩小边框或折叠页面元素。

1️⃣ 第一次使用，按这四步走

  1. 双击运行 start.bat。脚本会自动打开受控浏览器，默认使用项目内的 chrome_profile/；第一次启动通常是空白、未登录状态。
  2. 在受控浏览器中打开目标 AI 网站并完成登录，直到进入真正可以对话的页面。以 Gemini 为例，建议打开 https://gemini.google.com/，并保持左侧侧边栏处于展开状态。
  3. 正式发请求前，请确认受控浏览器里只保留目标 AI 网站，不要留下搜索页、邮箱、教程页等无关标签页，以免干扰脚本自动操作。
  4. 回到客户端，按下方「连接 API」填写地址、密钥和模型名；随后再打开控制面板 http://127.0.0.1:8199/ 查看日志和配置即可。

💡 想跳过重新登录？这属于高级选项。普通使用更推荐继续使用项目自带的 chrome_profile/，把登录态留在这个独立目录里，隔离性最好。

⚠️ 只有在明确理解浏览器用户目录结构和会话数据风险时，才建议复用现有登录态副本。如果你确实需要这样做，请只复制你自己的本地浏览器目录副本，不要共享、分发或混用他人的登录态数据。详细步骤放在后面的「浏览器常量」章节中。

🧭 站点配置

  • 选择器：告诉程序“输入框、发送按钮、结果容器”分别在哪。
  • 工作流：决定操作顺序，比如先新建对话、再填充输入，还是直接回车发送。
  • 响应检测：决定什么时候算回复结束。
  • 多模态提取 / 文件粘贴：分别对应图片、音频、视频资源回传，以及长文本附件场景。
  • 预设：同一站点可以拆多套配置，适合“日常对话 / 识图 / 长文本 / 代码”等不同用途。

🗂️ 标签页池、日志、设置与提取器排查

  • 标签页池：查看所有标签页的编号、状态、当前 URL、请求计数；也可以复制指定标签页的 API 端点或切换预设。
  • 提取器：当前主要作为站点预设中的配置字段使用；当“网页明明有回复，但抓出来内容不对”时，再按本页后面的提取器章节排查。
  • 日志：定位问题到底卡在发送前、等待中还是提取阶段；如果你觉得 DEBUG 原文太硬，也可以在设置里打开 INFO / DEBUG 日志可爱化。
  • 命令：为异常状态、周期检查和自动恢复动作配置触发器与执行逻辑。
  • 设置：管理环境配置、浏览器常量、AI 元素识别等全局参数。

建议的阅读顺序：第一次使用时先看“站点配置”和“标签页池”；排查失败时优先看“日志”；准备接新站点时再看“设置 → AI 元素识别”。

📊 性能监控与版本管理

控制面板内置了高级运维监控和版本回退/切换机制，以便开发者实时观察本地服务的健康度以及进行版本管理。

1. 请求监控 (Request Monitor)

点击控制面板的「请求监控」选项卡，可查看当前服务的实时运行性能与统计数据：

  • 系统占用 (Memory Usage)：显示后台浏览器及 Python 服务的实时内存占用。如果在高并发下出现异常，可优先监控该项以检查 Chrome 内存泄漏情况。
  • 硬盘状态与请求总量：展示当前磁盘剩余空间以及服务自启动以来的累计网络请求吞吐总量。
  • 全局请求成功率：实时计算最近数百次请求的统计成功率（成功 vs 失败占比），以便判断当前本地网络或 AI 站点防线的稳定性。
  • 分站点成功率统计：按域名聚合请求，统计例如 gemini.google.com 或 arena.ai 的单独请求健康度。
  • 历史请求列表：显示最近请求的详细执行链。支持点击特定行查看完整上下文，并标有“多模态”等专属交互类型标签，便于进行交互追踪。

请求失败时怎么查

  1. 先进入「请求监控」，看最近一条失败记录的域名、标签页编号、耗时、错误摘要和请求类型。
  2. 点开该记录查看详情，确认失败发生在排队、输入、发送、网络监听、DOM 回退还是媒体后处理阶段。
  3. 再切到「日志」，用请求 ID 或页面上显示的短追踪号串联同一轮请求的 [ROUTE]、[POOL]、[INPUT]、[STREAM]、[NetworkMonitor] 日志。
  4. 如果同站点失败率突然升高，优先检查受控浏览器是否触发验证码、登录过期、页面改版或代理 IP 切换。
  5. 如果某个标签页长时间保持忙碌，先在标签页池里查看当前任务；确认确实卡死后，再使用调试接口取消或强制释放。

后端历史接口：

  • GET /api/system/request-history?limit=200：读取最近请求历史。
  • GET /api/system/request-history/{request_id}：读取单条请求详情；前端详情弹窗使用的就是这个接口。
  • POST /api/debug/cancel-current：DEBUG 模式下取消当前运行请求，可选 tab_id。
  • POST /api/debug/force-release：DEBUG 模式下强制释放请求锁和标签页锁，只建议排查卡死时使用。

日志隐私说明：输入粘贴验证、文件粘贴和请求摘要会尽量记录长度、哈希、阶段和状态，而不是把完整原文长期打印在日志里。需要更深排查时再临时打开 DEBUG，并在问题确认后及时关闭。

2. 版本管理 (Version Management)

点击「版本管理」部分，可以免去手动下载解压的繁琐步骤：

  • 版本拉取列表：系统自动联网获取 GitHub Releases 发布的历史版本（包含发布时间与更新说明链接）。
  • 一键切版本/更新：可以直接点击对应历史版本行的「切换到此版本」按钮。系统会启动自动更新器（updater.py），在下载升级前，更新器会自动对您的本地核心代码和静态资源进行快照备份。一旦更新失败，它将自动执行回滚，保证本地环境不被损坏。

🆕 新增站点

新增站点有两条路线：让系统自动识别，或者手动新增并自己配置。建议第一次先走自动识别，跑通后再人工修正。

路线 A：AI 自动识别

  1. 进入 控制面板 → 设置 → 环境配置，填好 HELPER_API_KEY、HELPER_BASE_URL、HELPER_MODEL。
  2. 在脚本控制的浏览器里打开目标站点，并停留在真正可聊天的页面。准备发起请求前，请确认受控浏览器里没有除了目标网站之外的其它网站或无关标签页。
  3. 对该站点发起第一次真实 API 请求。
  4. 如果这个域名还不在 config/sites.json 中，后端会自动抓取 HTML 并调用辅助 AI 做页面分析。
  5. 分析成功后，会自动生成主预设并写入站点配置文件。

关键点：自动识别会在未知域名第一次实际请求时启动。点“新增站点”按钮只会先建一个空壳；只打开页面、不发请求时，后端不会自动写入配置。

✅ 如何确认识别成功：

  • 日志页会出现 [AI分析] 字样的条目，最终显示"分析完成"或"配置已写入"。
  • 刷新控制面板左侧站点列表，能看到该域名已被添加。
  • 进入该站点配置页，检查选择器是否已被自动填写——务必人工校对 result_container，这是最容易识别偏差的字段。
  • 如果域名已出现但选择器全是空的，通常说明辅助 AI 无法解析该页面（可能是未登录、停在欢迎页，或页面 HTML 被混淆），需要手动配置选择器。

路线 B：手动新增并自己配置

  1. 点击左下角 新增站点。
  2. 输入域名，例如 chat.example.com。
  3. 进入“主预设”，先补三个最小核心选择器。
  4. 再补最短工作流并逐个测试。
  5. 最后保存并实际调用一次 API。

🔌 连接 API

本项目提供兼容 OpenAI 格式 的接口，绝大多数支持“自定义 OpenAI 端点”的客户端都能直接使用。

⚙️ 客户端通用配置参数：

  • 服务提供商 (Provider)：选择 OpenAI、OpenAI 兼容 或 自定义。不同客户端的叫法不完全一样，选那个能填写自定义地址的即可。
  • API 接口地址 (Base URL)：
    • 默认（自动分配标签页）：http://127.0.0.1:8199/v1
    • 指定站点域名：http://127.0.0.1:8199/url/gemini.com/v1（自动匹配该站点的标签页）
    • 指定标签页：http://127.0.0.1:8199/tab/1/v1（编号见下方「标签页池」）
    • 指定当前完整 URL：http://127.0.0.1:8199/tab-url/{token}/v1（token 从「标签页池」里复制）
    • 部分客户端需要完整路径：在末尾补上 /chat/completions
  • API 密钥 (API Key)：默认未启用认证时，填一个占位值即可（例如：sk-local）。
    本项目默认不会校验这个值是否来自上游站点；唯一例外是你开启了「认证」功能（AUTH_ENABLED=true），这时它必须和 .env 中的 AUTH_TOKEN 保持一致。
  • 模型名称 (Model)：建议填写一个便于客户端保存配置的占位名称（例如 web-api 或 gemini-web）。实际响应来源仍由你当前打开的站点和预设决定。

✅ 测试步骤：把上面的地址、密钥和模型名填好后，直接发一句正常的话进行测试即可；不建议只点客户端里的“测试 API”按钮。

指定预设的三种方式

现在聊天接口支持直接指定 preset_name。这不会改变站点默认预设，只对当前这一次请求生效。

方式 A：直接写进路径

curl "http://127.0.0.1:8199/url/gemini.com/pro/v1/chat/completions" ^
  -H "Content-Type: application/json" ^
  -d "{\"model\":\"any\",\"messages\":[{\"role\":\"user\",\"content\":\"你好\"}],\"stream\":false}"

方式 B：放到 URL 查询参数

curl "http://127.0.0.1:8199/url/gemini.com/v1/chat/completions?preset_name=pro" ^
  -H "Content-Type: application/json" ^
  -d "{\"model\":\"any\",\"messages\":[{\"role\":\"user\",\"content\":\"你好\"}],\"stream\":false}"

方式 C：放到请求体

{
  "model": "any",
  "messages": [
    { "role": "user", "content": "你好" }
  ],
  "stream": false,
  "preset_name": "pro"
}

优先级规则：如果三处都传了预设，则按 路径里的预设 > URL 查询参数 > 请求体 的顺序生效。

💡 实战建议：如果客户端允许你自定义 Base URL，最推荐直接写成 http://127.0.0.1:8199/url/gemini.com/pro/v1 这种路径风格；如果客户端不方便改 Base URL 结构，再退回 ?preset_name=pro 或请求体字段。

响应头诊断

路由类接口会尽量返回诊断响应头，方便你确认后端实际采用了哪种分配策略。

响应头含义
X-Tab-Selection-Modefirst_idle / round_robin / random / tab_index / exact_url 等实际选择模式
X-Requested-Route-Domain域名路由请求的目标域名；动态路由尚未选定标签页时也会返回
X-Resolved-Tab-Index已经明确解析到固定标签页时返回的标签页编号
X-Resolved-Route-Domain已经明确解析到标签页时，该标签页当前所属域名
X-Resolved-Exact-Url精确 URL 路由最终命中的完整 URL
X-Resolved-Preset-Name路径或请求参数最终命中的预设名称

动态域名路由说明：当你调用 /url/{domain}/... 且没有传 tab_index 时，后端会把 selector 作为分配模式传给标签页池，真正的标签页会在工作流开始执行时获取。因此这类响应可能只有 X-Requested-Route-Domain 和 X-Tab-Selection-Mode，不会提前伪装成已经解析到了某个标签页。

💡 关于登录状态

  • 推荐登录：建议在网页上登录你自己的账号，这样本地接口才能继承当前会话本来就能使用的站点能力与历史上下文。
  • 免登录也可以：如果该网站在未登录状态下允许对话，则无需登录也能直接通过 API 调用。

⚠️ 关于 SillyTavern（酒馆）：酒馆里的“测试 API”按钮发送的请求通常过短，部分站点不会正常响应。建议直接发送一句对话进行测试，而不是依赖测试按钮。

⚠️ 上下文长度限制：受限于网站输入框一次性所能容纳的字符上限，上下文不能太长，否则请求会直接失败。对于长文本场景，可以开启「文件粘贴」功能，让站点改按附件流程处理内容。

  • Gemini 官网：无会员约 30k 字符；Pro 会员目前没有明确上限。
  • Arena AI：约 120k 字符。

🧰 函数调用（Tool Calling）

项目兼容 OpenAI 标准 tools 格式，也兼容 functions / function_call

在网页模型输出这一侧，后端现在优先引导项目自己的 XML 调用块：<adapter_calls> / <call> / <arg>。旧的 <tool_calls> / <invoke> / <parameter> 仍然兼容。

后端内置了 函数调用校验失败后的内部修复重试，可以直接在 控制面板 → 设置 → 环境配置 → 函数调用 里切换策略、控制重试次数和结果长度上限。

这是什么：兼容 OpenAI 标准的 tools 调用格式，让你继续用熟悉的方式定义工具。

默认输出外壳：<adapter_calls> → <call name="..."> → <arg name="...">

注意：这不是原生 API 的函数调用。实际工作方式是：后端把工具定义、工具历史和调用约束整理成提示词 -> 网页端模型按格式输出 -> 后端再尝试把结果解析回 OpenAI 风格的 tool_calls。所以它的稳定性核心取决于模型自己的理解能力，而不是官方 API 层的强制执行。

预填充乱序零宽设置 → 环境配置 → 函数调用开启后，额外的预填充和尾部提示词会随机乱序，并插入少量零宽字符。

内部修复重试次数设置 → 环境配置 → 函数调用控制自动修复后再试多少轮；设为 0 相当于关闭内部修复。

完整上下文模型老是修不对，怀疑缺上下文把原对话和修复反馈一起发回去，信息更全，但也更容易把模型注意力重新拉散。

典型失败表现

  • 模型直接输出自然语言，而不是工具调用
  • XML 或 JSON 结构不完整，后端无法解析
  • 参数名对不上，或字段层级错位
  • 混合输出：解释文字 + 半个工具调用

最重要的预期管理：如果你遇到“明明该调工具却直接说了一段话”“参数对不上”“后端提示无法解析结构化调用块”这类现象，先优先怀疑网页端模型这次没能稳定产出 <adapter_calls> 或有效 JSON 回退，客户端本身往往没有接错。

实战建议

  1. 先只给 1 到 3 个工具测试，不要一上来塞十几个。
  2. 函数名尽量用清晰的英文动词，比如 search、calculate、get。
  3. 参数 schema 尽量扁平，先避免超过两层的深层嵌套。
  4. 优先选择更强、更听话的模型；模型不够强时，规则写再多也可能失败。
  5. 如果解析总是失败，先简化工具和参数，再考虑换模型。

🗂️ 标签页池

标签页池是本项目的核心调度机制，理解它就理解了多标签页并行的原理。

工作流程

  1. 你在受控浏览器中打开一个或多个 AI 网站标签页。
  2. 脚本自动检测并为它们分配编号（1、2、3……）。
  3. API 请求到达时，自动挑一个空闲标签页处理。
  4. 处理完成后，标签页会释放回池中，等待下一个请求。

五种路由方式

方式地址格式适合场景
自动分配 /v1/chat/completions 不在意具体用哪个标签页，让系统自动挑空闲的
指定站点 /url/gemini.com/v1/chat/completions 同时开了多个不同站点，想确保请求发到固定站点
指定标签页 /tab/1/v1/chat/completions 为不同标签页配了不同预设，需要精确控制
标签页 URL 路由 /tab-url/{token}/v1/chat/completions 只匹配当前完整 URL，不回退到同域名其它页面；token 从标签页池面板复制
URL 绑定预设路由 /tab-url/{token}/{预设}/v1/chat/completions 严格匹配完整 URL 后，再严格使用该站点的指定预设

💡 在控制面板里你可以：

  • 查看所有标签页的编号、状态、当前 URL 和请求计数。
  • 复制默认路由、域名路由、固定标签页路由、标签页 URL 路由和预设专属路由。
  • 为不同标签页切换不同的预设配置。
  • 启用或禁用域名路由、固定标签页路由、精确 URL 路由等路由方式，减少不想暴露的入口。

标签页分配模式

模式含义适合场景
first_idle 优先选择编号最靠前的空闲标签页；没有空闲时再等待 默认模式，行为最可预测
round_robin 在匹配站点的标签页之间轮询 同一站点开了多个标签页，想均匀分摊请求
random 在匹配站点的标签页之间随机选择 不希望请求总落到固定编号标签页时使用

默认值怎么来：域名路由没有显式传 selector 时，会跟随标签页池面板里的全局分配模式；显式传了 ?selector=random 或 ?selector=round_robin 时，只影响当前这一次请求。

域名路由的高级参数

参数示例作用
selector ?selector=round_robin 在匹配站点的多个标签页之间轮询、随机或优先空闲
tab_index ?tab_index=2 在域名路由基础上进一步锁定某个标签页
preset_name ?preset_name=pro 本次请求临时覆盖预设，不改标签页绑定和站点默认值

💡 推荐理解方式：/url/gemini.com/... 负责“去哪个站点”，selector / tab_index 负责“选哪个标签页”，preset_name 负责“这一轮用哪套配置”。三者是可以组合的。

精确 URL 路由什么时候用

  • 同一个站点打开了多个不同会话 URL，并且你必须把请求发回其中一个固定会话。
  • 域名相同但页面用途不同，例如一个标签页是普通对话，另一个标签页是图像生成或代码工作区。
  • 你需要让客户端配置一个“不会因为同域名其它标签页空闲而跑偏”的 Base URL。

⚠️ URL 变化提醒：/tab-url/{token} 基于完整 URL 生成。部分站点在新建对话或发送第一条消息后会把 URL 从首页跳到会话页；这时旧 token 可能不再匹配，需要在标签页池里重新复制最新的 URL 路由。

⚠️ 注意事项：

  • 标签页编号在单次运行期间保持不变，重启脚本后会重新分配。
  • 关闭某个标签页后，对应编号会自动释放；新开的标签页通常会在几秒内被检测到。
  • chrome://newtab 等空白页不会被识别，页面加载完成后才会纳入池中。
  • 如果所有标签页都在忙，新请求会自动排队等待。

🎛️ 预设系统

预设系统允许你为同一个站点创建多套独立的配置方案，并将不同预设分配给不同的标签页。

标签页预设名称配置差异

如何使用预设

1. 创建预设

  1. 在控制面板中选择一个站点
  2. 在「配置」选项卡顶部可以看到预设选择器
  3. 点击「＋ 新建预设」，输入名称后创建
  4. 新预设会克隆当前选中预设的所有配置
  5. 然后可以独立修改新预设的选择器、工作流、提取器等配置
  6. 如需让该预设成为默认，点击「⭐ 设为默认」

2. 为标签页分配预设

  1. 切换到「标签页」选项卡
  2. 在对应标签页的行中找到「预设」下拉框
  3. 选择想要使用的预设即可

3. 通过 API 使用站点域名路由或指定标签页

# 假设当前已有一个 Gemini 站点标签页，标签页 #2 使用"快速识图"预设

# 使用 Gemini 站点标签页
curl http://127.0.0.1:8199/url/gemini.com/v1/chat/completions

# 使用固定编号的快速识图标签页
curl http://127.0.0.1:8199/tab/2/v1/chat/completions

4. 在单次请求里临时覆盖预设

# 方式 A：路径指定预设（最适合直接配置成 Base URL）
curl "http://127.0.0.1:8199/url/gemini.com/pro/v1/chat/completions" ^
  -H "Content-Type: application/json" ^
  -d "{\"model\":\"any\",\"messages\":[{\"role\":\"user\",\"content\":\"帮我写一段介绍\"}],\"stream\":false}"

# 方式 B：查询参数指定预设
curl "http://127.0.0.1:8199/url/gemini.com/v1/chat/completions?preset_name=pro" ^
  -H "Content-Type: application/json" ^
  -d "{\"model\":\"any\",\"messages\":[{\"role\":\"user\",\"content\":\"帮我写一段介绍\"}],\"stream\":false}"

# 方式 C：请求体指定预设
curl http://127.0.0.1:8199/url/gemini.com/v1/chat/completions ^
  -H "Content-Type: application/json" ^
  -d "{\"model\":\"any\",\"messages\":[{\"role\":\"user\",\"content\":\"帮我写一段介绍\"}],\"stream\":false,\"preset_name\":\"pro\"}"

# 固定标签页 + 临时指定预设
curl "http://127.0.0.1:8199/tab/2/v1/chat/completions?preset_name=pro" ^
  -H "Content-Type: application/json" ^
  -d "{\"model\":\"any\",\"messages\":[{\"role\":\"user\",\"content\":\"继续上一轮\"}],\"stream\":false}"

调用规则：显式传入 preset_name 时，本次请求会强制使用该预设；如果路径、查询参数、请求体都传了，则按路径 > 查询参数 > 请求体 生效。不传时，仍然按“标签页绑定预设 → 站点默认预设”的旧规则执行。

💡 提示：每个预设包含独立的选择器、工作流、流式配置、多模态提取、文件粘贴等全套配置。修改一个预设不会影响其他预设。未手动指定时会自动使用站点默认预设（default_preset）。

💬 对话复用与新建会话控制

为了让您在本地测试时能够灵活选择是“在现有网页会话上继续聊天”还是“强制重新开始新对话”，预设配置中提供了以下精细化控制（见控制面板中每个预设的「对话复用」控制区）：

  • 对话复用窗口 (秒)：
    • 当该值大于 0 时，同一个标签页在此时间窗口（秒）内收到后续请求，会直接复用当前网页的会话上下文进行续聊。
    • 设为 0 则表示关闭对话复用（默认值），每次请求都可能根据工作流设定触发“新建会话”逻辑（如果工作流开头配置了点击 new_chat 动作）。
  • 强制新建对话：
    • 开启此开关后，即使您配置了大于 0 的对话复用窗口，系统也会忽略复用设定，**始终在每次请求前强制新建会话**，这特别适合测试无历史干扰的独立 Prompt 流程。

🔍 选择器配置 (Selectors)

每个站点（的每个预设）都需要定义一组 CSS 选择器，告诉程序去哪里输入、哪里点击。

零基础先做这 3 件事

先认 3 个核心字段

第一轮只盯住 input_box、send_btn、result_container。这 3 项填对了，页面通常就能完成最基本的收发测试。

input_box 输入位置 send_btn 发送入口 result_container 回复区域

先去可视化工作台走一遍

本地页面现在不只是演示，而是一个真正能用的选择器工作台。你可以切换目标字段、修改页面状态、查看推荐候选、验证动态 class 会不会把你的写法搞崩。

打开选择器工作台 
"selectors": {
  "input_box": "textarea[id='prompt']",   // 输入框
  "send_btn": "button.send-btn",          // 发送按钮
  "result_container": ".markdown-body",   // AI 回复所在的容器
  "new_chat_btn": "button.new-chat",      // 新建对话按钮
  "temp_chat_btn": "button.temp-chat"     // 临时对话按钮 (自定义)
}
  • input_box: 必填。聊天输入框。
  • send_btn: 必填。发送按钮。
  • result_container: 必填。包含 AI 回复内容的容器。
  • new_chat_btn: 选填。每次新会话开始时点击的按钮。

💡 怎么找得更稳：

  • 优先看 textarea、button、input[type=file]、aria-label、data-testid 这类信息。
  • 如果 class 名很像随机生成的一长串字母数字，先别急着用，后面页面一改就可能失效。
  • result_container 尽量选整条 AI 回复的外层。只选到某个 p 或 span 时，漏内容的概率会明显变高。

填写顺序建议

  1. 先右键输入框，检查元素，把 input_box 填进去。
  2. 再找发送按钮，把 send_btn 填进去。
  3. 让页面出现一条 AI 回复，找到整条回复的外层，把 result_container 填进去。
  4. 每填一项就点一次「测试」，确认它能命中，别攒到最后一起查。
  5. 前三项稳定之后，再看要不要补 new_chat_btn、message_wrapper、上传相关字段。

新的可视化工作台怎么用

  1. 先在右侧选择你要练的字段，左侧模拟页面会直接高亮真正目标。
  2. 把你自己的选择器输进去，看它是“唯一命中正确目标”“命中了但太宽”，还是“命中了别的元素”。
  3. 直接查看下方的推荐候选，优先挑 唯一命中 的写法继续复测。
  4. 点一次「刷新随机 class」，确认你的写法不是靠动态类名碰巧命中。
  5. 如果命中了多个元素，就去看“命中元素详情”，先搞清楚它到底选到了谁，再回头收敛。

🧪 测试选择器：控制台里的测试按钮现在已经升级成“选择器测试工作台”了，不只是告诉你找到了没，还会返回命中详情、推荐候选、风险提示，并且可以直接把候选回填到当前字段。建议每填一项就顺手点一次测试。

💡 自定义选择器：你可以自己新增选择器，比如新增一个「临时对话按钮」，然后在工作流里添加点击这个按钮的步骤，程序就会在执行时多点一步临时对话。

🧩 提取器配置 (Extractors)

当 AI 回复“抓回来不干净、缺内容、丢格式”时，通常先看提取器；如果问题不在网页 HTML，而在底层响应体，那就继续往下看解析器和网络拦截。

// 在站点配置中为预设配置提取器
{
  "extractor_id": "deep_mode_v1",  // 使用深度模式
  ...
}
  • 默认模式: 直接从 result_container 中提取文本，适合大多数普通对话场景。
  • deep_mode (深度模式): 对复杂代码块、LaTeX 公式等内容做额外处理，适合对格式要求更高的场景。

💡 配置建议：如果网页上已经有完整内容，只是提取结果比较乱，那就先留在提取器里排查。默认模式不够用时，再切到 deep_mode；如果问题在于“内容根本还没渲染出来”，或者你想直接读取底层 JSON / 文本，那就该去看解析器和网络拦截模式了。

⚠️ 复杂内容提醒：即使是 deep_mode，某些复杂代码块也可能依然翻车。如果你追求更完整的代码块、公式或原始流式输出，可以继续看下方「响应检测配置」里的网络拦截模式。不过它需要站点已经有对应解析器，未适配的站点不要随意开启。

🎞️ 多模态提取

控制面板里这个功能现在叫多模态提取。它统一负责从网页中提取图片、音频、视频，并把结果整理成下游前端能直接使用的本地 URL / Markdown 引用。

"image_extraction": {
  "enabled": true,                // 兼容旧字段；实际会跟随内部模态自动同步
  "modalities": {
    "image": {
      "enabled": true,
      "run_policy": "on_signal",
      "quick_probe_timeout_seconds": 1.0,
      "late_wait_timeout_seconds": 45
    },
    "audio": {
      "enabled": true,
      "run_policy": "probe_if_trigger_found",
      "quick_probe_timeout_seconds": 1.0,
      "capture_timeout_seconds": 12
    },
    "video": {
      "enabled": true,
      "run_policy": "on_signal",
      "quick_probe_timeout_seconds": 1.0,
      "late_wait_timeout_seconds": 90
    }
  },
  "selector": "img",              // 图片元素选择器
  "audio_selector": "audio, audio source",
  "video_selector": "video, video source",
  "container_selector": ".img-grid", // (可选) 限定在特定区域查找
  "audio_capture_enabled": false, // 是否允许点击朗读按钮并录制页面播放音频
  "download_blobs": true,         // 是否下载 blob/src 或远程媒体到本地
  "mode": "all",                  // all / first / last
  "max_size_mb": 10               // 最大允许大小
}

兼容性说明：当前配置键名仍然是 image_extraction，旧版 "image": true 这种布尔写法仍会被自动迁移；新教程建议使用上面的策略对象，因为它可以把“支持某种模态”和“是否执行长等待 / 录音探测”分开。

运行策略怎么选

策略含义适合场景
disabled完全关闭这个模态。站点不需要图片 / 音频 / 视频时使用。
generic_only只做一次快速 DOM 扫描，不触发长等待、朗读点击或录音。Gemini 直连提取这类“可能返回媒体，但不想每轮卡住”的预设。
on_signal先快速扫一遍；只有发现明确生成信号时才进入长等待或后续动作。图片 / 视频生成，或者解析器能上报媒体生成状态的网站。
probe_if_trigger_found音频专用。快速找朗读按钮，找到才尝试点击和捕获。有朗读按钮但不希望每轮都硬等音频的网站。
always_probe只要模态启用就主动探测。确认每轮都会产出对应媒体，且愿意接受等待成本时再用。

什么是 signal

这里的 signal 指“值得继续等媒体”的证据，而不是单纯“站点支持这个模态”。常见信号包括：当前回复 DOM 已经出现媒体节点或占位节点；解析器上报 media_generation_state.pending；回复文本里出现生成中 / pending / audio / video 等占位提示；请求明显是图片生成；或者音频策略快速探测到了朗读入口。

💡 关键变化：现在流程会先做 quick_probe_timeout_seconds 控制的快速扫描。只有策略和 signal 允许时，才会进入 late_wait_timeout_seconds 的长等待或 capture_timeout_seconds 的音频录制。也就是说，网站“支持音频/视频”不再等于每一轮都长时间等待。

界面上的使用方式：

  • 先开启顶部的“多模态提取”总开关，再在图片、音频、视频卡片里分别选择启用状态和运行策略。
  • 图片和视频主要调 长等上限；音频主要调 快速探测 与 录制上限。
  • Gemini 这类直连媒体预设建议让音频 / 视频保持 generic_only，并关闭“页面播放音频捕获回退”。

文件保存位置：

  • 你发送给 AI 的图片: 保存在根目录 image/。
  • AI 生成 / 返回的图片、音频、视频: 统一保存在根目录 download_images/，并返回可访问的本地路径。

💡 返回效果：图片会以内嵌 Markdown 图片形式返回；音频和视频会以类似 [audio_0](/download_images/xxx.mp3)、[video_0](/download_images/xxx.mp4) 的链接形式返回，方便其他前端直接消费。

💡 使用建议：先确认你真正需要哪一种模态，再给它选择最保守可用的策略。只想捞现成 DOM 媒体时用 generic_only；需要等待生成结果时再用 on_signal；只有稳定有朗读按钮的网站才开启音频捕获。

🔊 页面朗读音频捕获

除了直接提取网页里的 audio / video 节点，系统也可以点击页面里的“朗读 / 收听 / TTS”按钮，把 AI 播放出来的语音音频抓回来。这条链路由 audio_capture_enabled 单独控制，不会因为音频模态启用就必然执行。

"image_extraction": {
  "modalities": {
    "audio": {
      "enabled": true,
      "run_policy": "probe_if_trigger_found",
      "quick_probe_timeout_seconds": 1.0,
      "capture_timeout_seconds": 12
    }
  },
  "audio_capture_enabled": true,
  "audio_trigger_selector": "button[aria-label=\"朗读\"]",
  "audio_trigger_labels": ["朗读", "语音朗读", "收听"],
  "audio_network_capture": {
    "enabled": true,
    "transport": "page_websocket_probe",
    "url_patterns": ["voicegenie"],
    "extractor": "voicegenie_ogg_pages",
    "timeout_seconds": 2.5,
    "settle_seconds": 0.35
  }
}

工作顺序：系统会先等待文本回复完成，再按音频策略做快速触发入口探测。点击成功后，优先从页面内网络流（如 WebSocket TTS 帧）直接提取音频；如果网络直抓不够完整，才回退到页面播放录音。

关键配置说明：

  • audio_capture_enabled：是否启用“点击朗读并抓音频”这条链路。Gemini 直连提取预设默认应保持关闭。
  • audio_trigger_selector：优先使用的朗读按钮选择器。站点按钮稳定时，建议直接填精确 selector。
  • audio_trigger_labels：当 selector 不稳定时，用这些按钮文本做兜底匹配。
  • audio_network_capture.enabled：是否优先尝试网络直抓。
  • audio_network_capture.url_patterns：只捕获 URL 命中这些关键字的音频网络流。
  • audio_network_capture.extractor：网络流聚合器。当前豆包朗读使用 voicegenie_ogg_pages。

豆包朗读预设：项目里已经给 www.doubao.com 的朗读预设接好了这套能力，默认策略是：探测到朗读入口才执行，网络直抓优先，页面录音兜底。

试听说明：如果你希望页面能正常外放试听，需要保证受控浏览器本身不是用 --mute-audio 启动的。单纯刷新网页无法解除浏览器进程级静音，必须重启 9222 那个受控 Chrome 实例。

常见现象排查：

  • 页面音频捕获未触发播放按钮：通常是朗读按钮选择器过时，或者页面当前根本没出现朗读入口。
  • tracked_media=0, tracked_web_audio=0, chunks=0：说明按钮点到了，但这轮播放链没有被页面捕获脚本接管。
  • 偶发截断：当前会优先尝试网络直抓；如果网络音频看起来过短或不完整，系统会丢弃它并回退到页面录音，避免把半截音频返回给前端。

🌊 响应检测配置

系统需要知道 AI 什么时候开始输出、什么时候算真的说完。这里提供两种模式：默认的 DOM 模式，以及更进阶的网络拦截模式。

什么时候该看网络监听？

大多数时候先留在 DOM 模式

页面里的回复能稳定抓到时，DOM 模式就够用了。第一次适配新站点时，也建议先把 DOM 模式跑通，定位问题会直接很多。

遇到这些情况再切网络拦截

代码块总是缺、公式总是乱、网页里看得到内容却很难完整提取，或者站点走的是底层 XHR / Fetch 接口时，这时再切到网络拦截更合适。若站点本身是整包 JSON / 文本返回，它依然会表现成一次性完成。

listen_pattern 请求关键词 parser 响应解析器
模式是否流式说明
DOM 模式（默认推荐） ✅ 流式 像“盯着网页看”一样检测页面变化，适合绝大多数普通对话场景
网络拦截模式 ✅ 流式（按站点适配） 直接拦截浏览器网络请求并解析响应体，适合追求更稳的复杂内容提取

DOM 模式（默认推荐 👀）

DOM 模式像“用眼睛盯着网页内容看”一样工作：只要页面内容还在变化，就说明 AI 还在继续输出。它最适合纯文本对话、角色扮演和大多数日常聊天场景。

  • 适用场景：绝大多数普通文本对话。
  • 优势：通用性强，不需要针对每个站点单独写解析器。
  • 局限：依赖网页 HTML 结构，复杂代码块和公式有时会抓不全。

检测逻辑

生成结束判定 = (连续稳定次数 ≥ 稳定判定次数) AND (静默时间 > 静默超时阈值)

工作原理：

  1. 脚本按设定间隔反复检查页面内容。
  2. 如果内容还在变，说明 AI 仍在输出，系统会重置稳定计数并继续流式返回新增内容。
  3. 如果内容没再变化，稳定次数增加，静默时间继续累加。
  4. 当“稳定次数”和“静默时间”同时达到阈值时，系统判定这一轮回复结束。
  5. 如果一直等到「全局硬超时」都还没看到内容出现，通常说明 result_container 选错了，或 AI 还没真正开始回复。

网络响应解析（进阶推荐 📡）

⚠️ 前提条件：这个模式必须配合站点解析器使用。没有适配的站点不要随意开启，否则通常既抓不到有效数据，也会明显增加排障复杂度。

网络拦截模式会观察与当前会话相关的 XHR / Fetch 响应，再把命中的 JSON 或文本转换成最终输出或增量流。对于已适配站点，它通常更快、更稳，对复杂代码块和公式的还原也更好。

  • 适用场景：追求更完整的代码块 / 公式，或者网页本身不适合从 DOM 提取内容。
  • 优势：对已适配站点通常更稳，流式输出更接近原始响应。
  • 前提：必须有可用的 parser，并配好合适的 listen_pattern。

💡 如何开启：在控制面板的站点配置 -> 「流式配置」中，把模式切换为「网络拦截」即可。

💡 上手顺序：先抓对请求关键词，再选解析器，最后再调超时。很多人卡在这里，是因为一上来就调秒数，真正的问题还留在 listen_pattern 和 parser 上。

维护者 / 高级调试步骤

  1. 用受控浏览器打开目标站点，停在真正聊天的页面。
  2. 打开开发者工具的 Console。
  3. 把项目附带的 static/拦截请求发生器.txt 脚本完整复制进去回车运行。如果第一次打开 Console 还不能直接粘贴，就先用鼠标点一下控制台输入区域，或者先手动输入一次 allow pasting（部分浏览器内核会要求先解除粘贴保护），再重新粘贴脚本。
  4. 手动发一条最简单的消息，让站点真实请求完整跑一遍。
  5. 请求结束后，在 Console 里执行 exportRequests()，浏览器会下载导出的请求 JSON。
  6. 把这个 JSON 连同 app/core/parsers/base.py、app/core/parsers/__init__.py 和一两个相近解析器一起发给 AI。
  7. 让 AI 新建 app/core/parsers/xxx_parser.py，并告诉你 __init__.py 里需要怎么注册。
  8. 回到控制台，开启网络拦截模式，填上 AI 告诉你的 listen_pattern 和 parser ID。
  9. 开始测试；如果结果不对，把“导出文件 + 错误现象 + 新 parser 文件”继续发给 AI 微调。

💡 发给 AI 时最好明确这几件事：

  • 这是“非流式响应”，请优先从完整响应体里直接提取最终文本。
  • 第一次成功解析后就应该返回 done=True，避免重复追加内容。
  • 如果包含图片，请填到 images 字段里。
  • 请告诉我需要修改 app/core/parsers/__init__.py 的哪些代码来完成注册。

"workflow": [

  { "action": "WAIT", "value": 0.5 },

动作类型说明

FILL_INPUT在输入框填入用户问题

📊 命令优先级与执行顺序

  • 每条命令可设置 priority（任意整数，默认 2），数字越大优先级越高。
  • 系统请求基准优先级由 CMD_REQUEST_PRIORITY_BASELINE 控制（默认 2）。
  • 当命令优先级高于基准点时，命令会优先于新请求获取标签页执行。
  • 高优先级命令若包含 clear_cookies，会等待同域忙碌标签页结束后再执行，降低对并行会话干扰。

💡 环境变量示例：在 .env 中设置 CMD_REQUEST_PRIORITY_BASELINE=2 作为系统请求基准优先级，修改后需重启服务。

🧳 想沿用自己的浏览器登录态，应该怎么配？

这是一个高级选项。如果你希望脚本启动后直接继承自己平时浏览器里的登录状态、Cookie 和扩展，请使用 独立复制出来的用户数据目录，不要直接把脚本指向你日常正在使用的系统默认目录。

⚠️ 重要：从 Chrome 136 开始，Chrome 对远程调试做了安全调整，默认 Chrome 数据目录不再接受 --remote-debugging-port。如果你把 BROWSER_PROFILE_DIR 直接写成系统默认的 User Data，浏览器可能会被打开，但调试端口不会真正起来，脚本也就无法接管它。Chrome 官方说明见：Changes to remote debugging switches to improve security。

⚠️ 会话数据提醒：复制浏览器用户目录时，里面可能包含登录态、Cookie、本地缓存和扩展数据。只建议复制你自己的本机目录副本，并且只用于本机调试；不要把这类目录打包分享、提交仓库或发给别人。

推荐做法：复制一份专用用户目录
  1. 先完全关闭所有 Chrome 窗口。
  2. 新建一个专用目录，例如 C:\Users\你的用户名\AppData\Local\UniversalWebApiProfile。
  3. 从原始目录 C:\Users\你的用户名\AppData\Local\Google\Chrome\User Data 中，至少复制 Local State 和你要使用的配置子目录（如 Default、Profile 1）到这个新目录。
  4. 在 .env 中把脚本指向这份副本，然后重新运行 start.bat。
BROWSER_PATH=C:\Program Files\Google\Chrome\Application\chrome.exe
BROWSER_PROFILE_DIR=C:\Users\你的用户名\AppData\Local\UniversalWebApiProfile
BROWSER_PROFILE_NAME=Default

字段含义：

  • BROWSER_PATH：浏览器程序路径。
  • BROWSER_PROFILE_DIR：用户数据根目录，应该指向 User Data 这一层，或者你复制出来的专用目录。
  • BROWSER_PROFILE_NAME：具体配置名称，常见值是 Default、Profile 1、Profile 2。

容易写错的地方：BROWSER_PROFILE_DIR 不要写成 ...\User Data\Default。正确写法是：

BROWSER_PROFILE_DIR=C:\Users\你的用户名\AppData\Local\Google\Chrome\User Data
BROWSER_PROFILE_NAME=Default

也就是说，BROWSER_PROFILE_DIR 写“根目录”，BROWSER_PROFILE_NAME 写“具体配置名”。

如果你不在乎沿用自己的登录态：最省心的方式是留空 BROWSER_PROFILE_DIR，让项目继续使用根目录下的 chrome_profile/。这样隔离最彻底，也最不容易和日常 Chrome 互相影响；对大多数用户来说，这也是默认推荐方案。

🛠️ 更新白名单

在控制面板 → 设置 → 更新白名单中修改。点击「保存」后会写入 config/update_settings.json，并在下次自动更新时生效。

  • 作用：它控制的是“更新时是否保留不变”，不是运行时启用/禁用开关。
  • 目录项：勾选 config/、app/、static/ 这类目录项时，会保留整个目录。
  • 合并更新：config/sites.json 与 config/commands.json 在未勾选时默认会按更新器规则合并，尽量保留本地站点、命令和状态，同时补入发布版新增内容；如果勾选，则整份文件保持本地版本不动。
  • 内部保留项：config/update_settings.json 会被更新器自动保留，无需手动勾选。

当前默认保留项：

  • config/sites.local.json：站点默认预设覆盖
  • config/commands.local.json：命令启用状态与分组覆盖
  • chrome_profile/：浏览器登录态、Cookie 与用户目录，仅建议本机自用，不建议共享或提交到仓库
  • venv/：Python 虚拟环境
  • logs/：日志目录
  • image/：图片输出目录
  • updater.py：更新脚本本体
  • .git/：Git 元数据目录
  • __pycache__/：Python 缓存目录
  • *.pyc：Python 编译缓存文件
  • backup_*/：更新前备份目录

可勾选范围覆盖核心配置、本地覆盖、运行数据、后端代码、前端资源、脚本与文档、开发文件。只想保留登录态和个人配置时，通常保持默认勾选即可；需要保留自己改过的代码时，再额外勾上对应目录。

⚠️ 警告：目前可视化编辑器没有撤销功能！一旦保存无法快捷复原，请务必谨慎修改。

❓ 常见问题

Q1: 酒馆显示「连接失败」怎么办？

A: 酒馆的「测试 API」按钮发送的请求通常太短，部分站点不会正常响应，尤其是 AI Studio 这类站点更容易误报失败。建议不要点测试按钮，直接发送一句真实对话来测试。

Q2: 请求频繁失败，怎么排查？

A: 可以按下面这个顺序逐一排除：

  1. 脚本运行时操作了受控浏览器：工作流执行过程中不要点击、切换或操作受控浏览器页面，最多只能看着。
  2. 缩小了窗口或折叠了页面元素：浏览器可以最小化，但不要手动缩小窗口边框，也不要折叠脚本需要操作的页面区域，否则脚本可能找不到元素。
  3. 受控浏览器里保留了无关标签页：只保留目标 AI 网站。搜索页、邮箱、资讯页或其它普通网页都可能干扰自动化；如果已经开着并出现异常，重启脚本即可。
  4. 网站本身状态异常：切到受控浏览器看一下，是否触发了人机验证、被内容风控拦截，或者因为单次输入太长而被网站拒绝。
  5. 网络波动或代理切换了 IP：这类情况可能导致发送失败甚至卡死，表现为后台显示发送成功，但浏览器没有真正发出去或发出后又失败。可以等待约 5 分钟，或直接重启脚本。
  6. 站点配置已经过期：如果前面都排除了，可能是站点改版导致选择器或配置失效。可以去 GitHub 提 Issue、加群反馈，或者自己手动更新选择器配置。

Q3: 同时打开多个标签页，脚本会怎么处理？

A: 脚本会通过标签页池自动调度这些标签页：

  • 默认路由：/v1/chat/completions，系统自动挑一个空闲标签页处理。
  • 指定站点：/url/gemini.com/v1/chat/completions，只把请求发到对应站点的标签页。
  • 指定标签页：/tab/1/v1/chat/completions，精确使用编号为 1 的标签页。
  • 你也可以给不同标签页配置不同的预设，例如一个专门聊天，一个专门识图。
  • 如果同时来了多个请求，空闲标签页会依次分配；全部忙碌时，新请求会排队等待。

Q4: 标签页池里看不到新打开的标签页？

A: 请先检查这几项：

  • 页面是否已经加载完成；chrome://newtab 这类空白页不会被识别。
  • 等待 2 到 3 秒后，点击控制台里的「立即刷新」。
  • 如果还是不出现，直接重启脚本。

Q5: 控制面板里的参数需要手动调整吗？

A: 大多数时候不需要。默认值已经覆盖了最常见的使用场景，遇到具体问题时再针对性调整即可。

Q6: 目前有哪些已知问题？

A: 目前确认过的已知问题主要包括：

  • VSCode 的 Codex 插件及 Claude Code 均已完全适配（自 2026-05-28 起）。具体连接方法可参见「连接 API」章节。
  • DrissionPage 每次升级后，都需要重新执行对应补丁。
  • “独立 Cookie 标签页”目前仍属于高级实验功能。教程正文暂不展开这部分用法；如果强行追求“同一受控浏览器窗口内的独立 Cookie”，在浏览器原生菜单等场景下可能出现稳定性问题，当前建议优先使用共享 Cookie 方案。

Q7: 代理切换命令不生效怎么办？

A: 可以按下面这几项检查：

  1. 确认 Clash 正在运行：并且已经启用了 External Controller。
  2. 确认 API 地址正确：默认一般是 http://127.0.0.1:9090，有些客户端会用别的端口。
  3. 确认代理组名称拼写正确：访问 http://127.0.0.1:9090/proxies 看实际名称。
  4. 确认 Secret 已填写：如果 Clash 配了 secret，命令配置里也要一起填。
  5. 查看日志：在控制台「日志」里搜索 [CMD] 开头的条目，能看到具体报错或切换结果。

Q8: 想复用自己的 Chrome 登录态，为什么配了还是不生效？

A: 最常见的原因有这三个：

  • 直接指向了系统默认的 User Data：Chrome 136+ 上这样做很容易导致远程调试端口起不来。请先复制一份独立目录，再让脚本使用那份副本。
  • BROWSER_PROFILE_DIR 层级写错了：它应该指向 User Data 这一层，而不是 ...\User Data\Default。具体用哪个子配置文件夹，请写在 BROWSER_PROFILE_NAME。
  • 改完配置没有重启脚本：浏览器启动相关配置只有重新运行 start.bat 才会生效。

具体步骤可以回看前面的浏览器登录态复用说明。
